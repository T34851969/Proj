## 任务分配

为保证两周内交付并降低沟通成本，现将每位成员的具体任务明确如下（合并到 main 前请通知组长并确认本地/单元测试通过）。

- **组长**: 负责最终合并与集成验收、维护 `main` 分支、搭建基础环境（Docker、仓库结构、`deploy.sh`）、日常协调与里程碑把控。
- **后端 — AI 工程师**: 负责提示词设计与评估、模型接入抽象层（DeepSeek/Qwen）、路由/降级策略、成本控制与生成质量监控；提供模型测试接口与示例 prompt。
- **后端 — 服务层工程师**: 负责 FastAPI 接口实现、数据库表与迁移脚本、API 文档、鉴权与输入校验、与 AI 层的对接及示例接口。
- **后端 — 控制组件工程师**: 负责异步任务与 Worker 实现（Celery/Redis）、任务调度、重试/回退、Markdown→HTML→PDF/DOCX 的渲染流水线与日志/监控埋点。
- **前端（1 人）**: 负责表单、模板选择、上传/预填充、在线预览与与后端接口的联调。可优先使用 Python 全栈方案以统一语言（团队同意时），若选择 React 等技术亦可。
- **模板与 UI 设计**: 负责至少 5 套简历模板的视觉设计与可打印样式（响应式 & 打印一致性）、样本数据填充、导出兼容性测试与模板文档。
- **测试/运维（轻量）**: 负责编写 Dockerfile 与 docker-compose、手动部署脚本（deploy.sh）、基本单元/集成测试（pytest）、并在必要时执行压力测试（Locust 简版）与运行验证。

> 说明：两周优先保证端到端核心流程（表单→AI生成→预览→PDF导出）可运行；较大变更（数据库迁移、模板大调整）需提前通报。

## 团队协作例程（必须遵守）

### 1) 仓库与分支策略

- 仓库以组长持有的 GitHub 仓库为中心，开发遵循组长指定的主仓库地址。
- 开发模式：持续在 `main` 分支上开发（传统集成模式）。要求：
  - 提交前在小组内沟通改动范围与影响；
  - 小步提交、每次改动范围尽量小；
  - 必要时（大变更、可能破坏兼容）先通知组长。

### 2) 提交信息规范

- 推荐格式：`type(scope): 简要说明`，可选 body 描述详细变更。
  - 示例：`feat(api): add create_resume endpoint`
  - type 可选：`feat`、`fix`、`docs`、`chore`、`refactor`、`test`。
- 提交须关联 Issue（若存在），在提交描述中引用 Issue 编号。

### 3) 合并与代码审查（merge）

- 由于采用 `main` 始终开发策略：
  - 每次提交前请自行运行单元测试与基本集成测试。
  - 提交后在团队群/项目看板说明变更点与风险点。
  - 确认无误后才能最终合并；合并前需满足“合并清单”（见下）。

### 4) Issue 与任务管理

- Issue 模板（简短）：
  - 标题：`[类型] 简短描述`
  - 内容：背景、复现/需求、预期、负责人、预计截止时间（可选）。
- 建议 Label：`enhancement`、`bug`、`infra`、`docs`、`urgent`、`blocked`。
- 每位成员可自由领取任务，但承担任务者需在 Issue 中说明预计进度与阻塞点。

### 5) 合并清单（合并前必须检查）

- 单元测试通过（`pytest`）
- 关键功能手动联调通过（尤其后端 API 与前端交互）
- 若含 DB 变更：提供 `Alembic` 迁移脚本并说明回滚方法
- 文档更新（README / API 文档 / 模板说明）
- 代码风格通过 lint（可选）
- 已在团队群通报改动范围与合并时段

### 6) 测试规范

- 单元测试：`pytest`，尽量覆盖后端核心逻辑（生成流程、RAG 检索、渲染）。
- 集成测试：局部联调，确保 API + Worker 的基本链路可执行。
- E2E（可选）：使用 `Playwright` 做关键路径验证（生成并下载 PDF）。

### 7) 部署与运行（占位，实际等组长下发）

- 本地构建示例：

```bash
# 构建与运行后端（示例）
docker build -t resume-backend:dev -f backend/Dockerfile .
docker run --rm -p 8000:8000 --env-file .env resume-backend:dev

# 本地启动 celery worker（开发环境）
celery -A app.worker worker --loglevel=info
```

- 提交到组长机器手动部署（示例脚本 `deploy.sh`，由运维/测试负责人维护）：

```bash
./deploy.sh --env production
```

### 8) 密钥与安全

- 严禁将 API Key、私钥、密码提交到仓库。
- 在项目根目录使用 `.env` 文件（加入 `.gitignore`），并在 `README` 中说明需设置的环境变量清单。
- 存储敏感用户信息时（PII）务必加密或使用字段脱敏。

### 9) 沟通与会议

- 每日在项目群中汇报进展。
- 任何阻塞请立即在群内 @ 相关人员并创建 Issue 标注 `blocked`。

### 10) 时间线与验收标准

- 两周目标：完成表单→AI 生成→预览→PDF 导出的端到端流程。
- 验收：核心路径能在组长提供的物理机或本地容器上手动运行并导出可用 PDF。

### 11) 如何请求合并（操作步骤）

1. 在团队群/看板说明改动与合并时间窗口。
2. 在 Issue/PR 中列出验证步骤与注意点。
3. 运行本地测试并截图或记录运行结果。
4. 在仓库打临时 tag（如有必要）。
5. 通知组长进行最终合并。
